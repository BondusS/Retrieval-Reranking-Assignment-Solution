# Решения задачи поиска статей справки

Представляю описание архитектуры и методологии решения, разработанного для ранжирования статей справки по пользовательским запросам. 

Решение построено в виде следующего пайплайна: 
* Парсинг и очистка текста
* Составление фичей представляющих различные меры релевантности 
* Ранжирования на их основе за счёт ranker модели бустинга.

С таким подходом удалось достичь результата `MAP@10 = 0.56` на тестовом датасете.

---

## Обработка HTML-разметки статей

Статьи даны в исходном виде с HTML-разметкой, таблицами и другими служебными элементами. 
Для их очистки и приведения к текстовому формату используется следующий подход:

* **Очистка от тегов:** С помощью библиотеки `BeautifulSoup` (парсер `html.parser`) из документов извлекается чистый текст.
* **Удаление лишних пробелов:** С помощью регулярного выражения `re.sub(r'\s+', ' ', text)` удаляются повторяющиеся пробелы, 
    символы табуляции и переноса строк, оставшиеся после удаления тегов разметки.
* **Приведение к нижнему регистру:** Очищенный текст приводится к нижнему регистру для унификации перед подачей в текстовые модели.

Так получается чистый текст, пригодный для статистического анализа (BM25, TF-IDF) и кодирования векторными моделями.

## Используемые признаки, модели и алгоритмы

В основе решения лежит двухэтапный пайплайн: сначала извлекаются разнообразные лексические и семантические признаки, 
а затем модель градиентного бустинга объединяет их для принятия финального решения.

### Используемые модели

* **Bi-Encoder (Семантика фрагментов):** `intfloat/multilingual-e5-large`. Модель ориентирована на асимметричный поиск. 
    Справляется сопоставлением коротких пользовательских вопросов и длинных фрагментов текста с использованием
    префиксов (`query: ` для запроса, `passage: ` для документов).
* **Bi-Encoder (Семантика документа):** `BAAI/bge-m3`. Модель с поддержкой длинного контекста (до 8192 токенов). 
    Используется для получения комплексного семантического представления статьи целиком.
* **Лексические модели (Точные совпадения):**
    * `BM25Okapi` из библиотеки `rank_bm25` для классического ключевого поиска.
    * `TfidfVectorizer` из `scikit-learn` для построения символьных и словарных матриц совпадений.

### Набор из 10 извлекаемых признаков

Для каждой пары `(Запрос, Статья)` рассчитывается вектор из 10 признаков, представляющих различные меры релевантности:

1.  `bge_score`: Косинусное сходство между вектором запроса и вектором статьи целиком (BGE-M3).
2.  `bge_title`: Косинусное сходство между вектором запроса и вектором заголовка статьи (BGE-M3).
3.  `e5_max`: Максимальное косинусное сходство среди всех перекрывающихся чанков статьи (E5-Large).
4.  `e5_mean`: Среднее косинусное сходство по всем чанкам статьи (E5-Large).
5.  `e5_title`: Косинусное сходство между вектором запроса и вектором заголовка статьи (E5-Large).
6.  `bm25_title`: Оценка BM25 для лемматизированного заголовка статьи. Лемматизация выполняется с помощью `pymorphy3` с кэшированием результатов.
7.  `bm25_body`: Оценка BM25 для лемматизированного тела статьи.
8.  `tfidf_char`: Косинусное сходство символьных n-грамм (диапазон длин от 3 до 5). Защищает модель от опечаток пользователей.
9.  `tfidf_word`: Косинусное сходство лемматизированных слов и словосочетаний (униграммы и биграммы).
10. `q_length`: Длина запроса пользователя в словах. Позволяет бустингу корректировать веса признаков в зависимости от детальности вопроса.

### Алгоритм ранжирования
Для агрегации признаков используется модель **`CatBoostRanker`** с функцией потерь **`YetiRank`**. 
Этот алгоритм оптимизирует порядок выдачи документов напрямую, ориентируясь на метрику ранжирования (MAP). 
YetiRank оценивает пары документов внутри одной группы (запроса), что делает обучение более стабильным по сравнению 
с поточечными (pointwise) методами классификации.

## Валидация качества на calibration.f

Для оценки надежности и обобщающей способности решения применяется строгая схема валидации:

* **K-Fold Кросс-Валидация:** Данные из `calibration.f` разбиваются на 5 фолдов по группам запросов (`query_id`). 
    Это гарантирует, что запросы из обучающей выборки никогда не попадают в валидационную выборку в рамках одного фолда.
* **Разделение логики обучения и валидации:** 
    * *Обучение:* Чтобы модель фокусировалась на сложных границах решений, применяется метод **Hard Negative Mining**. 
      Датасет для обучения формируется только из Top-75 кандидатов на запрос (отобранных по быстрой базовой эвристике) 
      и истинно релевантных статей (Ground Truth). Это сокращает объем тривиального шума в обучающей выборке.
    * *Валидация (Inference):* Оценка качества модели на валидационном фолде проводится в условиях, 
      максимально приближенных к реальному поиску - предсказания рассчитываются для **всех 793 статей** базы справки.
* **Итоговая метрика:** Качество оценивается по значению **`MAP@10` (Mean Average Precision at 10)**. 
    На текущей конфигурации модель демонстрирует честный Out-Of-Fold результат `MAP@10 = 0.57` (с отдельными фолдами, достигающими значения `0.6+`).

## Анализ ошибок и внесенные изменения

В процессе разработки решения был проведен анализ расхождений между предсказаниями моделей и разметкой в `calibration.f`. 
Были локализованы следующие типы системных ошибок, и предприняты меры по их устранению:

* **Размытие семантики в длинных текстах (Semantic Dilution):**
  * *Проблема:* Семантический вектор статьи, содержащей много разнородных разделов, усреднялся, из-за чего точечные ответы на узкие вопросы не находились.
  * *Решение:* Было внедрено разделение статей на перекрывающиеся фрагменты (чанки) по 120 слов с перекрытием в 40 слов. 
    В начало каждого чанка искусственно добавлялся заголовок статьи для удержания глобального контекста. 
    В качестве признаков стали использоваться `e5_max` (лучшее точечное совпадение) и `e5_mean` (общая тематика документа).
* **Слабая чувствительность к точным формулировкам заголовков:**
  * *Проблема:* Пользователи часто ищут информацию по ключевым фразам, вынесенным в заголовки статей, 
    но векторные модели на длинных текстах опускали приоритет заголовка.
  * *Решение:* Были рассчитаны отдельные признаки семантического сходства запроса непосредственно с заголовками статей (`e5_title` и `bge_title`). 
    Это помогло модели быстро выводить статьи с релевантными названиями на первые позиции.
* **Неустойчивость к опечаткам и разговорным сокращениям:**
  * *Проблема:* Присутствие в запросах опечаток (например, "кросовки", "заблокали") приводило к нулевым скорам в классическом BM25.
  * *Решение:* В пайплайн был добавлен символьный TF-IDF на n-граммах от 3 до 5 символов. 
    Он находит частичные совпадения в корнях слов даже при наличии серьезного шума в написании.
* **Искажение намерений (Intent Erasing) из-за очистки стоп-слов:**
  * *Проблема:* В ранних версиях проводилась слишком агрессивная очистка запросов от вопросительных местоимений ("как", "где", "почему"). 
    Из-за этого модель путала статьи-инструкции ("как вернуть товар") со справочными статьями о правилах сервиса.
  * *Решение:* Из списка стоп-слов были убраны все вопросительные и Domain-специфичные слова. 
    Теперь удаляется только чистый вежливый шум ("здравствуйте", "подскажите пожалуйста"), сохраняя грамматическую структуру вопроса для нейросетей.
* **Засорение бустинга тривиальными негативными примерами:**
  * *Проблема:* YetiRank обучался на парах, где 99% документов не имели к запросу вообще никакого отношения. 
    Модель тратила ресурсы на разделение заведомо нерелевантных документов.
  * *Решение:* Переход к схеме Hard Negative Mining (обучение на Top-75 кандидатах от гибридного поиска) позволил сфокусировать 
    бустинг на тонких различиях между похожими статьями, что дало наиболее заметный прирост итоговой метрики.
